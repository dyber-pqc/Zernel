How We Built an OS That Understands ML Workloads

By Zachary Kleckner · Dyber, Inc. · March 2026

Every ML platform today runs on top of a general-purpose operating system. The kernel doesn't know the difference between a data loading thread and a CUDA synchronization call. It can't tell when your GPU is starving for data. It doesn't know that NCCL all-reduce is the critical path in your distributed training.

We asked: what if the OS itself understood machine learning?

That question led us to build Zernel — a complete Linux distribution where the CPU scheduler, memory manager, network stack, and observability layer all understand ML workloads natively.

The Insight: ML Has Predictable Phases

If you profile any ML training job, you'll see a repeating pattern:

[Data Loading] → [Forward Pass] → [Backward Pass] → [All-Reduce] → [Optimizer Step]

Each phase has completely different system requirements:

  • Data Loading: CPU-intensive, I/O-heavy, GPU idle. The CPU should be running data workers at maximum priority.
  • GPU Compute: GPU at 100%, CPU mostly idle. The CPU scheduler should yield immediately — every cycle wasted here delays the next data batch.
  • NCCL Collective: Network-heavy. The network stack should prioritize NCCL traffic above everything else.
  • Optimizer Step: Brief CPU burst. Should preempt everything to minimize GPU idle time.

The standard Linux scheduler (CFS) treats all these phases identically. It has no concept of GPU utilization, NCCL collectives, or data pipeline starvation. It just round-robins CPU time according to process niceness.

The Technical Approach: sched_ext + eBPF

Linux 6.12 merged sched_ext — a framework for writing fully custom CPU schedulers as BPF programs that load into the kernel at runtime. This is the exact capability we needed: a way to replace the CPU scheduler without forking the kernel.

Zernel's scheduler is written in Rust (userspace daemon) + C (BPF kernel program). The userspace component maintains per-task state and runs phase detection. The BPF component reads scheduling decisions from a shared map and applies them at microsecond granularity.

Phase Detection

We classify each ML process into one of five phases using these heuristics:

DataLoading:    io_wait > 30% AND gpu_utilization < 10%
GpuCompute:     gpu_utilization > 80% AND cpu_burst = 0
NcclCollective: nccl_shm active AND futex_wait_count > 0
OptimizerStep:  cpu_burst < 5ms AND last_gpu_sync recent
Unknown:        default (CFS-equivalent behavior)

Phase transitions are debounced with a stability counter to prevent scheduling flap.

NUMA-Aware CPU Selection

On multi-socket GPU servers, the scheduler reads NUMA topology from /sys and pins tasks to CPUs on the same NUMA node as their GPU. This eliminates cross-socket memory access latency for data transfers.

Phase-Aware Power Management

This is where the energy savings come from. During the DataLoading phase, the GPU is mostly idle — there's no reason to run it at full clock speed. Zernel dynamically adjusts GPU clocks per phase:

DataLoading:    33% GPU clock, 100% memory clock → ~60% power
GpuCompute:     100% GPU clock, 100% memory clock → 100% power
NcclCollective: 50% GPU clock, 100% memory clock → ~70% power

On an 8xH100 cluster drawing 5kW, this saves 10-20% energy — roughly $4,000-8,000 per year per rack in electricity costs.

Beyond the Scheduler: 50+ Tools

The scheduler is Zernel's technical core, but the product needed to be more than a kernel patch. We built a complete terminal-native development environment: experiment tracking, GPU management, ML benchmarks, training debugger, model registry, distributed job orchestration, fleet management, compliance auditing, and post-quantum cryptographic protection for model weights.

Every tool is designed around one principle: zero code changes. Your existing PyTorch, JAX, and vLLM scripts run unchanged. Zernel observes and optimizes at the kernel level.

What We Learned

The hard part isn't the kernel work. sched_ext makes it surprisingly straightforward to write a custom scheduler. The hard part is building the complete product around it — the observability, the CLI, the packaging, the documentation, the testing — everything that turns a kernel feature into something a team would actually adopt.

eBPF is the right observability tool for ML. Traditional profiling requires code instrumentation. eBPF instruments at the kernel level — zero overhead, zero code changes, works with any ML framework. Our five probes (GPU memory, CUDA latency, NCCL, DataLoader, distributed sync) provide telemetry that no application-level tool can match.

Energy optimization is underrated. Everyone focuses on throughput (images/second, tokens/second). But at scale, energy costs are a significant part of GPU TCO. Phase-aware power management is a genuine competitive advantage that compounds over time.

Try It

Zernel is open-source (kernel components GPL-2.0, CLI proprietary). You can:

git clone https://github.com/dyber-pqc/Zernel.git
cd Zernel && cargo build --workspace --release
./target/release/zernel doctor
./target/release/zernel bench quick

Or download the bootable ISO at zernel.org and install it on your GPU server.
